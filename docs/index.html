Roadmap Studio

Deliverables by month, not a Gantt chart

This is a tool for showing what we intend to deliver, and roughly when — organized by workstream and by month. It is deliberately not a dependency chart, not a resource plan, and not a schedule you defend to the day.

What this is

A roadmap here is a grid: workstreams down the side, months across the top. Each cell holds the deliverables that workstream intends to land that month. That's the whole model.

It answers one question well — “by this month, what will exist that didn't before?” — and intentionally leaves the harder, lower-level questions (sequencing, effort, owners, risk) to the planning that happens after the roadmap, not inside it.

Why it isn't a Gantt

A Gantt chart encodes tasks, durations, and dependencies on a timeline. It's the right tool once the work is understood. But a roadmap is set before that — when you still owe yourself the discovery to figure out how the work decomposes. Forcing a Gantt too early invents precision you don't have yet, and that false precision becomes a commitment you didn't mean to make.

This board shows

  • Intended deliverables, per workstream
  • The month each is expected to land
  • The shape of a quarter at a glance
  • Milestones that gate a month

It deliberately omits

  • Task-level dependencies & critical path
  • Effort estimates / story points
  • Day-precise start & end dates
  • Resource allocation
The point

The roadmap is a statement of outcomes and timing. The plan is a statement of work. Keeping them in separate tools keeps each honest — the roadmap stays readable by anyone, the plan stays detailed enough to execute.

The method

The board is the top of a waterfall. You declare the deliverables and their target month first; the detailed planning flows down from there.

  1. State the deliverable. Write the outcome you want to exist, in the month you want it — one card per deliverable, in the right workstream. Keep it outcome-shaped (“PT & Ortho self-serve across core flows”), not task-shaped (“write the migration script”).
  2. Place it in time. Drag it to the month it should land. The column is the commitment; you're not picking a day, you're picking a month.
  3. Gate the big ones. Mark the months that carry a milestone. A milestone is a promise that a coherent set of deliverables is done — the natural checkpoint to re-plan against.
  4. Then waterfall down. For each deliverable, go work out how: break it into tickets, sequence them, find the dependencies, size the effort. That happens in your tracker (Linear, Jira), not here.
  5. Let reality push back up. When the lower-level plan says a deliverable can't make its month, you move the card — and the roadmap stays an honest reflection of intent, not a wish.

Anatomy of a board

  • Workstream — a durable line of work (Clinical, Workspaces, Automations…). Rows on the board.
  • Month — a column. The unit of timing. Coarse on purpose.
  • Deliverable (item) — a card in a workstream × month cell. The outcome that should exist by then. Supports markdown, nested sub-items, and an optional hard date when a specific day genuinely matters.
  • Milestone — a gate attached to a month: “by here, this set is done.”

Everything is a JSON you bring or build. The board is data; the app just renders and edits it.

From roadmap to plan

The handoff is the whole value. A deliverable on this board is an input to planning, not the plan itself:

Example

Card: **Rules UI complete** — build, test, promote, placed in Sept 1, in the Automations workstream.

That single card waterfalls into a dozen tickets — the builder, the draft model, ID/routing, dry-run, the promote flow — each with its own owner, estimate, and dependencies. None of that lives on the roadmap. The roadmap only says: this exists by September, and it's an Automations outcome.

The Gantt view — coming

Some audiences do want the timeline picture. A planned view toggle will let you flip the same board between two lenses without changing the data:

  • Deliveries (today) — the month-grid above. Each card is an outcome in its month.
  • Gantt (soon) — each deliverable becomes a bar; the delivery month is the bar's end, so a project reads as a span that finishes when the outcome lands.

Same JSON, same workstreams, two ways to read it — deliverables-first for planning, timeline-first for communicating.

Authoring

  • Edit mode — toggle it on, then drag cards between months, reorder within a column, add workstreams, and edit any text inline.
  • Markdown — card text renders live as you type. Right-click a card for bold / italic / code / strike / headings / lists.
  • Hard date — the ◆ on a card pins a specific day when the month isn't precise enough.
  • Bring your own JSON — drop a .json file anywhere to load a board; Export writes it back out in the same shape. Everything saves to your browser.
